# PaperPilot 利用説明書

この説明書は、PaperPilotを初めて使う研究者・学生・共同研究者向けです。論文を登録してから、根拠付きの質問、比較、整理、知識グラフの作成までを、現在の画面に沿って説明します。

> PaperPilotは研究判断を補助するツールです。回答、比較、研究ギャップ、仮説候補は、査読や原典確認の代わりにはなりません。重要な主張、数値、数式、前提条件は必ず引用元の原文で確認してください。

## 1. PaperPilotでできること

PaperPilotには、次の5つの主要画面があります。

| PC上部の表示 | 狭い画面での表示 | 主な用途 |
| --- | --- | --- |
| ライブラリ | 論文 | PDF・TXT・Markdownの登録、解析状態の確認、原本・抽出本文の閲覧 |
| 論文に質問 | 質問 | 登録論文を対象にした根拠付き質問、引用確認、研究対話の継続 |
| 比較・発見 | 分析 | 複数論文の目的・手法・結果・限界の比較、研究ギャップ候補の確認 |
| 研究ワークスペース | 整理 | タグ、検索履歴、保存した比較、文献情報のエクスポート |
| 知識グラフ | グラフ | 原典Source、アイデア、仮説、制約を根拠付きのノードとエッジで整理 |

PCでは上部ナビゲーション、画面幅が狭い場合は下部の「論文／質問／分析／整理／グラフ」で切り替えます。
以降は短い表示名を主に使います。PCでは上の表にある対応する長い名前を選んでください。

### PaperPilotが特に重視していること

- 原典とLLM生成物・研究者のアイデアを区別する。
- 回答の引用から、論文名、ページ、原文抜粋へ戻れるようにする。
- 削除ではなく状態を使って棄却した仮説や関係を残す。Edgeの状態変更は、変更者・理由・変更前後の状態を監査ログへ記録する。
- プロジェクトごとに論文、会話、ノート、タグ、グラフを分離する。

## 2. 最初の5分で行うこと

初めての場合は、次の順番がおすすめです。

1. 画面右上のプロジェクト名を確認します。
2. 「論文」でPDFを1〜3件登録します。
3. 各論文が解析完了になり、ページ数とチャンク数が表示されるまで待ちます。
4. 「質問」で、まず「この論文の目的・手法・結論を根拠付きでまとめて」と質問します。
5. 回答中の引用番号またはEvidenceカードを開き、原文を確認します。
6. 2件以上登録したら「分析」で比較します。
7. 継続して使う情報は「整理」のタグ・履歴、または「グラフ」の仮説ノードに残します。

## 3. プロジェクトと権限

### 3.1 プロジェクトとは

画面右上の「Project」に表示されている名前が、現在の研究プロジェクトです。論文、研究対話、検索履歴、タグ、ノート、保存した比較、知識グラフはプロジェクト単位で分離されます。

プロジェクト名を押すと、次の操作ができます。認証済みの利用者は、自分の現在のロールにかかわらず新しいプロジェクトを作成できます。

- 既存プロジェクトを選んで切り替える。
- 「新しい研究プロジェクトを作成」から新規作成する。
- owner権限の場合、鉛筆ボタンから名前を変更する。

新しいプロジェクトを作っても、現在の論文やノートは自動では移動しません。

### 3.2 権限

| 権限 | できること |
| --- | --- |
| owner | 閲覧・編集・削除、プロジェクト名・メンバー・権限の管理 |
| editor | 論文・質問・ノート・タグ・比較・グラフなどの作成と編集 |
| viewer | プロジェクトを閲覧し、質問画面で原文根拠を語句検索できる。LLM回答、会話への追記、追加、編集、削除は不可 |

ノートや検索履歴も個人専用ではなく、同じプロジェクトの共有データです。ownerはサイドバー（狭い画面ではヘッダー）の「メンバー」を開き、すでに同じ認証基盤へ登録済みのメールアドレスまたはユーザーIDを追加できます。そこでeditor/viewerへの変更と、owner以外のメンバー削除も行えます。最初のownerを外す・自分自身の権限を変える操作はできません。

## 4. 論文を登録する

### 4.1 ファイルから登録する

1. 「論文」を開きます。
2. 「ファイルを選んで追加」または、ドロップ枠内の「ファイルを選択」を押します。
3. `.pdf`、`.txt`、`.md` のいずれかを選びます。複数選択もできます。
4. 登録後、論文カードの進捗を確認します。

ドロップ枠へファイルを直接ドロップすることもできます。枠の空白部分をクリックしてもファイル選択は開かないため、ボタンを押してください。

標準設定の上限は次のとおりです。運用環境によって変更される場合があります。

- 一度に10ファイルまで
- 1ファイル25 MiBまで
- PDFは300ページまで

同じプロジェクトへ同一内容のファイルを再登録した場合、SHA-256で重複が検出され、既存の論文が使われます。

### 4.2 arXiv ID・DOIから登録する

1. 「arXiv / DOI」欄へIDを入力します。
2. 「メタデータ取得」を押します。

この操作は外部サービスからタイトル、著者、年、要約などを取得する機能です。原本PDFのダウンロード機能ではありません。取得できた要約は検索対象になりますが、ページ単位の引用確認が必要な場合は原本ファイルも登録してください。

### 4.3 解析状態の読み方

| 表示 | 意味 | 操作 |
| --- | --- | --- |
| 待機中 | 解析ジョブの開始待ち | そのまま待つ |
| 解析中 | 本文抽出・分割・埋め込みなどを処理中 | 進捗率を確認する |
| 完了 | 質問、比較、本文表示に使用可能 | 次の操作へ進む |
| 解析失敗 | 本文抽出などに失敗 | カードのエラーを確認し、原本を見直す |
| 登録済み | 同一ファイルが既に存在 | 既存のカードを使う |

解析中・解析失敗の論文は、質問対象、比較対象、本文ビューアで利用できません。

### 4.4 スキャンPDFとOCR

OCRは標準では無効です。画像だけで構成されたスキャンPDFは、本文を抽出できない場合があります。OCRを有効にした環境でも、数式、脚注、段組み、表の認識は完全ではありません。必ず原本と抽出テキストを比較してください。

## 5. 論文を選ぶ・読む

### 5.1 選択と閲覧は別の操作

論文カードでは、次の2つを区別します。

- カード左側のチェック: 質問・比較・エクスポートの対象に含める。
- 論文タイトルまたは「PDF・本文を開く」: 右側の根拠ビューアを開く。

何も選択していない場合、「質問」はプロジェクト内の解析済み全論文を検索します。

### 5.2 根拠ビューア

論文タイトル、回答中の引用番号、Evidenceカード、比較結果の論文名などから開けます。

根拠ビューアでは、次の情報を確認できます。

- PDF原本がある場合の原本表示
- ページごとの抽出テキスト
- 引用に対応するチャンクの強調表示
- 表、図、captionなどの抽出結果
- 「前のページ」「次のページ」による移動
- TXT・Markdownなどの場合の原本ファイル表示

右上の×または `Esc` で閉じます。ページを手動で移動すると、引用箇所の強調が解除される場合があります。

PDFが表示されなくても抽出テキストを利用できる場合があります。逆に、抽出テキストが不自然な場合はPDF原本を優先してください。

### 5.3 論文ノート

根拠ビューア内の「論文ノート」から、論文に紐づく共有ノートを作成できます。

1. ノートタイトルを入力します。タイトルは必須です。
2. 必要に応じて本文を入力します。
3. 「ノートを追加」を押します。

既存ノートは「編集」「削除」ができます。ノートは同じプロジェクトのowner/editorと共有されます。

## 6. 論文に質問する

### 6.1 基本操作

1. 「質問」を開きます。
2. 上部の検索範囲を確認します。
3. 「研究について質問」へ2文字以上、最大4,000文字で入力します。
4. 「質問する」を押します。
5. 回答とEvidenceを読み、引用元を開いて確認します。

候補ボタンも利用できます。

- 「論文全体から詳しくまとめて」
- 「前提の弱い部分を反証して」
- 「次の検証実験を設計して」

これらは質問欄への入力補助です。送信前に、自分の研究対象に合わせて具体化すると精度が上がります。

viewerは質問欄で「原文を検索」を使えます。この検索は選択した論文の語句一致からEvidenceを表示するだけで、埋め込み・LLM・SSE回答・会話履歴の保存は行いません。研究対話を作成したり、LLMによる回答を生成したりするにはownerまたはeditor権限が必要です。

### 6.2 検索対象を選ぶ

「プロジェクト知識ベース」を開くと、解析済み論文を個別にON/OFFできます。

- `N sources`: 選択したN件だけを検索します。
- `all N sources`: 解析済みの全論文を検索します。
- 選択解除: 全論文検索へ戻ります。

質問例:

- 「各論文で用いられたデータセット、評価指標、制約を表にして」
- 「定理Aが成立する前提条件を列挙し、引用ページを示して」
- 「論文Xと論文Yで結論が異なる原因候補を、原文と推測に分けて説明して」
- 「この仮説を反証する実験を3案出し、各案の根拠と不足情報を示して」

### 6.3 引用の読み方

回答中の `[1]` などの番号は、右側のEvidenceと対応します。Evidenceには通常、論文名、ページ、節、原文抜粋が表示されます。

引用番号の検査済み表示は、回答内の引用参照がシステム検査を通ったことを示します。論文そのものの正しさや、回答全体の科学的妥当性を保証するものではありません。

「LLM知識による補足」は登録論文ではなく、モデルの一般知識に基づく内容です。論文根拠として引用しないでください。

### 6.4 LLMフォールバック

OpenAI API、タイムアウト、レート制限、引用検査などの理由でLLM回答を使えない場合、検索した原文を使うローカル抽出回答へ切り替わります。これは根拠を保ったまま回答するための安全動作です。

画面に「LLMフォールバック」「根拠監査の注意」などが表示された場合は、Evidenceと原文の確認を優先してください。

### 6.5 研究対話と研究メモリ

- 「新しい研究対話」で新しい会話を始めます。
- 過去の会話を選ぶと、その続きから再開できます。
- 会話の要約や、仮説・前提・未解決点・次の検証候補が研究メモリとして残ります。

研究メモリは対話を継続するための要約であり、原典の証拠ではありません。

## 7. 論文を比較し、研究ギャップを探す

### 7.1 比較する

1. 「分析」を開きます。
2. 論文チップの `+` を押し、2件以上を選びます。
3. 「N件を分析」を押します。
4. 目的、手法、結果、限界・課題を比較します。
5. 必要に応じて名前を付け、「比較を保存」します。

選択対象を変更すると、表示中の分析結果はクリアされます。

比較は、要旨や本文断片から関連する文を抽出して並べる初期スクリーニング機能です。手法の意味的対応、数値単位の正規化、結論の妥当性評価までは行いません。各論文名から本文を開き、文脈を確認してください。

### 7.2 Research gaps

ギャップ候補は、本文中の `limitation`、`future work`、`課題`、`限界`、`今後` などの記述を手掛かりに表示されます。「該当ページを確認」から原文へ移動できます。

これは論文群の矛盾や未踏領域を自動証明する機能ではありません。表示されるOpportunityは検討を始めるための候補であり、検証済みの研究提案ではありません。

## 8. 調査結果を整理する

### 8.1 タグ

1. 「整理」を開きます。
2. 「タグと論文」で色とタグ名を指定します。
3. 「作成」を押します。
4. 各論文のチェックでタグを付け外しします。

鉛筆で名前・色を変更し、ゴミ箱で削除します。タグはプロジェクト単位です。

### 8.2 文献情報のエクスポート

「文献エクスポート」から次の形式をダウンロードできます。

- BibTeX
- RIS
- CSV

ライブラリで論文を選択している場合は選択中の論文、何も選択していない場合はプロジェクト内の全論文が対象です。タグでは対象を絞り込みません。

### 8.3 検索履歴

過去の質問、日時、対象論文を確認できます。

- 「再質問」: 当時の質問と論文範囲を使って質問画面を開く。
- 「削除」: 履歴を削除する。

### 8.4 保存した比較

分析画面で名前を付けて保存した比較を確認できます。比較名、論文数、結果の概要が保存されます。

## 9. 知識グラフを使う

知識グラフ画面は、左のSource、中央のGraph Canvas、右のInspectorという3列構成です。

### 9.1 最初に理解する用語

| 用語 | 意味 |
| --- | --- |
| Source Version | 原典テキストの不変な登録単位。本文ハッシュとLocatorを保持 |
| Source Span | Source内の引用可能な断片。ページ、行、セル、会話ターンなどの位置を保持 |
| Node | 原典事実、アイデア、制約、仮説など、グラフで扱う考えの単位 |
| Edge | 2つのNode間の関係。必ず1件以上のSource Spanを根拠に持つ |
| Reasoning Run | どのNodeを入力にし、どのNodeを出力したかを記録する推論来歴 |

Sourceは原典、Nodeはグラフ上の考えです。Sourceを登録しただけではNodeは自動作成されません。

### 9.2 Sourceを登録する

1. 左の「Source & Context」で形式を選びます。
2. 「出所・Locator」を入力します。
3. 本文を貼り付けます。
4. 「Sourceを登録」を押します。

対応形式:

- `latex`
- `python`
- `notebook`
- `csv`
- `chat`
- `markdown`

Locatorには、後から版を特定できる値を推奨します。

- コード: `repo://model.py@GitCommitSHA`
- Notebook: `repo://experiment.ipynb@GitCommitSHA`
- 会話: `chat://project-name/2026-07-13`
- メモ: `note://chaos-sync/hypothesis-v1`

本文はUTF-8で5 MiBまでです。登録時にブラウザとサーバーがSHA-256を照合し、形式に応じてSpanを抽出します。任意の文章を後からSource Spanとして追加することはできません。

PDFはここへ貼り付けず、「論文」画面から登録してください。現行のグラフ画面にはPDFアップロード操作がありません。ライブラリへ新しく登録し、本文抽出に成功したPDFは、対応するSourceが左の一覧に表示された場合に限り、そのページSpanを根拠として選べます。既存論文や抽出不能ページでSourceが表示されない場合は、グラフ画面だけで再生成することはできません。

### 9.3 Nodeを追加する

右下の「アイデアを追加」で型と本文を指定します。

| 型 | 用途 |
| --- | --- |
| Idea | 着想、検討候補、LLM提案 |
| Hypothesis | 検証可能な仮説 |
| Constraint | 数学的前提、物理制約、計算量、データ上の制約 |

追加されたNodeは `review_pending` になります。人間のレビューを経る前の生成物・メモであることを示します。

Source型Nodeは原典根拠を持つ事実用です。現行UIでは手動追加フォームにSource型はなく、バックエンドの根拠付き登録で扱います。

### 9.4 Nodeを選ぶ

- Nodeをクリックすると選択・選択解除できます。
- `Enter` または `Space` でも選択できます。
- 最後に選んだNodeの詳細がNode Inspectorに表示されます。
- 複数選択すると、直近2件がEdge作成のSource nodeとTarget nodeへ自動入力されます。

### 9.5 根拠付きEdgeを作る

1. Source nodeとTarget nodeを選びます。
2. Relationを選びます。
3. 「根拠Source」を選びます。
4. 表示されたSource Spanを1件以上チェックします。
5. 「根拠付きエッジを作成」を押します。

同じNode同士は接続できません。

| Relation | 使い方の目安 |
| --- | --- |
| `informs` | Source側の情報がTarget側の判断材料になる |
| `supports` | Source側がTarget側を支持する |
| `extends` | Source側がTarget側の理論・手法を拡張する |
| `formulates` | Source側の考えをTarget側で定式化する |
| `contradicts` | Source側とTarget側が矛盾する |
| `implements` | Source側がTarget側の仕様・理論を実装する |
| `depends_on` | Source側がTarget側を前提として依存する |
| `related` | 方向を強く定義しない一般的な関連 |

### 9.6 Edgeの状態を変更する

中央の線を選ぶと、Edge InspectorにStatus、Origin、Relationが表示されます。Originは関係が作られた経路を表し、`manual` は手動、`llm` はLLM、`import` はインポート由来です。owner/editorは、新しい状態と変更理由を入力して更新できます。

| 状態 | 意味 |
| --- | --- |
| `review_pending` | 未レビュー |
| `active` | 現在有効 |
| `verified` | 検証済み |
| `review_required` | 前提変更などにより再確認が必要 |
| `rejected` | 根拠不足・誤りとして棄却 |
| `superseded` | 新しい版・考えに置き換え済み |
| `pruned` | 現在の探索経路から枝払い |

状態変更には理由が必須で、監査ログとして保存されます。`rejected`や`pruned`は削除ではありません。バックエンドAPIのGraph検索では `active` と `verified` のNode・Edgeだけが対象になりますが、現行画面にはGraph検索を実行するボタンはありません。

### 9.7 現在の知識グラフの範囲

現行UIは、Sourceの登録、Nodeの手動追加、根拠付きEdgeの作成、Edge状態変更が中心です。次の機能はまだUIから利用できません。

- LLMによるNode・Edgeの自動生成
- 複数Nodeを選んだ順伝播・仮説生成
- 数式ASTの意味的同値検索
- 自動矛盾解消・自動逆伝播
- 思考ツリーの分岐・復元
- Graph検索結果のAskへの自動統合
- Reasoning RunのUI作成

## 10. データ保存と安全上の注意

### 保存されるデータ

- 原本ファイルと抽出アセット
- 論文メタデータ、ページ本文、検索チャンク
- タグ、論文ノート、検索履歴、保存した比較
- 研究対話、回答、引用、会話要約、研究メモリ
- Source Version、Source Span、Node、Edge、Edgeの状態遷移監査、推論来歴

Nodeの状態は現在値と更新日時を保存しますが、現行版ではNode状態変更の理由・変更者・変更履歴までは保存しません。理由付きの不変監査ログがあるのはEdgeの状態変更です。

標準のローカル構成では原本・派生ファイルは `backend/data/`、構造化データはPostgreSQLに保存されます。

OpenAI APIを利用する構成では、質問と選択された論文抜粋が回答生成のためOpenAIへ送信されます。機密情報を含む論文・会話を扱う場合は、組織のデータ取扱規則と運用設定を確認してください。

### 削除に関する注意

論文削除にはゴミ箱・復元機能がありません。削除すると原本、抽出アセット、ページ、チャンク、論文に紐づくSpanも削除されます。

知識グラフの根拠として使われている論文は削除できません。Edgeを `rejected` に変更しても根拠参照自体は残るため、現行版では運用者による対応が必要です。

## 11. よくある問題

### ファイルを追加できない

- PDF、TXT、Markdownのいずれかか確認する。
- ファイル数、サイズ、PDFページ数の上限を確認する。
- 空ファイルでないことを確認する。
- viewer権限でないことを確認する。
- 同一内容なら、既存論文として登録されていないか確認する。

### 解析中のまま進まない

- 画面を更新し、状態が変化するか確認する。
- 非同期worker構成では、RedisとCelery workerが動作しているか運用者に確認する。
- エラー表示がある場合は、その内容を運用者へ伝える。

### 本文が表示されない・文字化けする

- スキャンPDFでないか確認する。
- PDF原本を開き、テキストレイヤーがあるか確認する。
- OCR設定を運用者に確認する。
- 数式、段組み、表は抽出テキストだけで判断しない。

### 回答に引用がない・根拠が弱い

- 対象論文が解析完了か確認する。
- 上部の検索対象が意図した論文になっているか確認する。
- 論文中で使われている専門用語・表記に質問を合わせる。
- 質問を前提、対象、比較軸、希望する出力形式に分ける。
- 一般知識による補足と論文根拠を区別する。

### 論文を削除できない

知識グラフのNodeまたはEdgeが、その論文のSource Spanを根拠に使っている可能性があります。現行画面では根拠参照の削除機能がないため、運用者へ依頼してください。

### `Not Found`、401、403が表示される

- `Not Found`: 現在のプロジェクトに対象データがない、またはmembershipがない可能性があります。
- 401: 認証情報が失効している可能性があります。OIDC構成ではトークンを再設定します。
- 403: viewer権限など、操作権限が不足しています。

### LLM回答にならない

APIキー未設定、タイムアウト、レート制限、引用検査失敗などでローカル抽出回答へ切り替わることがあります。画面のLLM状態とフォールバック理由を確認してください。

## 13. 用語集

| 用語 | 短い説明 |
| --- | --- |
| Grounding | 回答や関係を、確認可能な原典へ結び付けること |
| Chunk | 検索に使うため論文本文を分割した断片 |
| Citation / Evidence | 回答の根拠になった論文、ページ、原文抜粋 |
| Hybrid search | 意味の近さと語句の一致を組み合わせる検索 |
| RAG | 検索した原文をLLMの回答材料として使う方式 |
| Provenance | 誰が、何を入力に、どの処理で生成したかという由来 |
| Locator | Sourceのファイル、版、日付などを特定する識別情報 |
| Immutable | 登録後に同じレコードの内容を書き換えず、新しい版として扱う設計 |
| review pending | 人間の確認前であることを示す状態 |
| Negative Knowledge | 棄却した仮説・関係・失敗経路を再発防止に使う考え方。現行版で理由付き履歴を監査できるのはEdge |
